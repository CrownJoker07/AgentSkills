# Interview Questions

按照"接收简历 → 总结简历 → 识别岗位 → 提取考察点 → 生成题目 → 输出到对话 → 写入飞书云文档"的顺序一气呵成。整个过程基于用户提供的原始简历内容，不持久化候选人信息。

## 强制触发与不反问

满足以下任一条件必须立即激活本 Skill：

- 用户发送 / 粘贴 / 上传了一份简历（文本或 PDF，含飞书消息附件）
- 用户消息中提到"简历 + 面试题 / 面试准备 / 考察点 / 模拟面试 / 飞书文档"
- 用户发来的材料明显是候选人简历（含教育、项目、技术栈等）

激活后一气呵成走完流程。禁止：

- 反问"是否需要生成面试题 / 是否写入飞书 / 是否继续下一步"
- 只回一段"已收到简历 + 核心信息罗列 + 是否需要我…"的确认式回复
- 只输出非结构化的简历总结或面试建议

第一次回复必须以 `## 一、候选人基本信息` 开头。仅在材料确实缺失必须阻断时才允许发问。

## 接收简历

三种输入方式，优先使用已有材料，不要同时要求多种：

- **粘贴文本**：用户直接粘贴简历全文。
- **本地 PDF**：用户提供本地 PDF 路径。使用 Read 工具读取，按页提取全文，保留段落结构。
- **飞书附件**：当前入口为飞书消息事件时，使用 lark-cli 读取原消息附件元信息并下载到本地临时路径，再用 Read 工具读取。

PDF 读取失败时说明原因并停止，不自行 OCR。只向用户请求最小替代，不要求整份重写。

敏感信息只保留与考察相关的内容（姓名、教育、经历、技能、项目、成果）。手机号、身份证号、邮箱、具体住址等不在输出中回显。

## 总结简历

在识别岗位和出题前，先基于原始简历做一份简要总结：

- 基本信息：姓名、教育背景、工作年限
- 岗位方向与当前定位
- 主要经历（倒序：时间 / 角色 / 核心职责）
- 代表性项目（2–4 个：目标 / 角色 / 技术栈 / 关键结果）
- 核心技能栈
- 可量化成果

总结用于呈现对齐，**后续步骤必须基于原始简历全文**，不得基于总结版本。

## 识别岗位方向

从原始简历推断岗位方向：后端 / 前端 / 客户端 / 全栈 / 算法 / 数据 / 测试 / 运维 / 产品 / 其他。推断必须引用原始简历中的具体依据；不充分时只向用户确认一次岗位方向，不根据猜测继续。

## 提取考察点

从原始简历抽取并归类：

- 核心技能
- 次核心技能
- 项目亮点
- 可疑点或待澄清点
- 成长轨迹

不得编造原始简历中未出现的信息。

## 生成题目

### 强制分类约束

固定六个二级标题，顺序固定，不得调换、合并、重命名或替换为自由分类：

- `### 4.1 动机与职业规划`
- `### 4.2 技术基础`
- `### 4.3 技术进阶`
- `### 4.4 项目深挖`
- `### 4.5 场景 / 设计`
- `### 4.6 软素质`

### 各分类定义

- **动机与职业规划**：为什么来面试该岗位、对该岗位的理解、实习目标、毕业去向与中长期职业规划、对目标城市 / 行业 / 团队风格的偏好。必须结合原始简历可推断的信息提问。**必问清单中的题目必须填入本分类第一题。**
- **技术基础**：核心技能的基本概念、原理和常用实现，贴近岗位方向。
- **技术进阶**：底层机制、性能、边界、设计取舍，难度高于基础题。
- **项目深挖**：针对原始简历中的具体项目，追问候选人的角色、方案选择、难点、关键数据、复盘。
- **场景 / 设计**：结合候选人经历给出一个业务场景或系统设计问题。
- **软素质**：协作、推动、复盘、抗压、目标管理等，必须引用原始简历中的具体段落。

不同岗位方向侧重不同，由模型按岗位特性自行调整各分类内题目的技术侧重。

### 面试时长约束

默认按 **3 分钟 / 题**出题，默认 **10 题 / 30 分钟**。题目总数 = 用户指定时长（分钟）÷ 3（四舍五入）；未指定一律按 10 题。

默认 10 题配额（合计 10）：

| 分类 | 配额 |
| --- | --- |
| 4.1 动机与职业规划（含必问题） | 2 |
| 4.2 技术基础 | 2 |
| 4.3 技术进阶 | 2 |
| 4.4 项目深挖（至少覆盖 2 个项目） | 2 |
| 4.5 场景 / 设计 | 1 |
| 4.6 软素质 | 1 |

题目总数变化时：

- \> 10：优先把多出的题分给 4.2 / 4.3 / 4.4（按顺序轮询），4.1 / 4.5 / 4.6 不动。
- < 10：优先从 4.2 / 4.3 / 4.4 依次各减 1；4.1 含必问题不得减到 1 以下，4.5 / 4.6 不得减到 0。

用户显式指定单题时长或总时长时，按用户节奏重新计算总题数，再按上述规则缩放。

### 出题优先级

1. **必问清单中的题目必须先填入对应分类**，计入配额，不得另外追加。
2. 在配额内优先选择与原始简历强相关的题目，不输出通用题库式问题。
3. 避免重复考查同一知识点。
4. 配额用满即停，不因简历信息密度高就突破总题数上限。

### 题目格式

每道题使用 Markdown 待办复选框格式：

```
- [ ] ★如果你入职后参与的项目方向与你最初预期不一致，你会如何衡量是否继续投入？
  考点：…
- [ ] 基于原始简历的其他动机 / 职业规划题
  考点：…
```

- 每道题以 `- [ ] ` 起行（方括号后一个空格）
- 必问题在 `- [ ] ` 之后紧跟 `★`，题目正文原样照抄必问清单
- 普通题目在 `- [ ] ` 之后直接写题目正文
- `考点：…` 独立成行，相对题目缩进 2 空格
- 不在题目之外输出参考答案、评分标准或追问清单，除非用户明确要求
- 不生成编程笔试或限时作业题，除非用户明确要求

## 写入飞书云文档

完成对话输出后，使用 lark-cli 将同一份骨架内容写入飞书云文档，每道面试题使用待办块 `[]`。

### 使用 lark-cli

优先使用已安装的 lark-cli Skills。能力名称或参数不确定时，先运行 `lark-cli schema` 或对应命令的 `--help`，再调用与飞书 OpenAPI 一一映射的接口；不要猜测快捷命令参数。

两类能力：

- **消息与附件读取**（`lark-im`）：飞书附件场景下读取原消息附件元信息并下载到本地临时路径。
- **云文档创建、写入与权限管理**（`lark-docx` 与文档成员能力）：将骨架内容写入飞书云文档，并在写入完成后将当前用户添加为文档协作者（`full_access`）。

结果作为 JSON 处理：以退出码和顶层 `ok` 判断成功，再从 `data` 读取 OpenAPI 结果；不要要求顶层存在原始 OpenAPI 的 `code: 0`。

### 文档创建与写入

1. 创建新文档。文档标题统一为 `面试题 - <候选人姓名或"未知"> - <岗位方向> - <YYYYMMDD>`。记录返回的 `document_id` 与文档链接。
2. 获取文档根 block_id：通常是新文档的 `document_id` 或返回结构中的根块标识，具体字段以 lark-cli 返回为准。
3. 按顺序批量创建块（一级标题用 `heading2`，二级分类用 `heading3`，不得调整顺序、不得合并、不得省略）：
   - `heading2`："一、候选人基本信息"
   - `text`：姓名、学校、专业、年级 / 届别、期望岗位、期望城市、实习时间、工作年限
   - `heading2`："二、简历总结"
   - `text`：岗位方向与当前定位、主要经历、代表性项目、核心技能栈、可量化成果
   - `heading2`："三、岗位匹配度分析"
   - `text`：岗位方向推断依据、关键亮点、潜在风险点、建议重点考察的方向
   - `heading2`："四、面试题清单"
   - `text`：`> 面试时长：<N 分钟>（<M 题 × 3 分钟/题>）`（与对话骨架首行一致）
   - 六个固定二级分类（顺序固定）：
     - `heading3`："4.1 动机与职业规划"
     - `heading3`："4.2 技术基础"
     - `heading3`："4.3 技术进阶"
     - `heading3`："4.4 项目深挖"
     - `heading3`："4.5 场景 / 设计"
     - `heading3`："4.6 软素质"
   - 每个分类下：
     - **每道面试题使用待办块 `todo`（`block_type = 17 / todo`），初始 `checked = false`**，块内首段为题目正文
     - 必问题在待办块正文开头加 `★` 标记（例如 `★如果你入职后参与的项目方向…`，不带序号）
     - 题目下方紧跟一个 `text` 块，内容为 `考点：…`，不塞进待办块正文
   - `heading2`："五、面试官备注"
   - `text`：信息不足的子项、需确认事项、建议追问方向
4. 一次批量请求中包含多块时按展示顺序排列；批量请求受 OpenAPI 单次上限限制时按顺序分批发送，保持原始顺序。
5. 文档写入完成后，将当前用户（即 lark-cli 当前登录身份，或环境变量 `LARK_MEMBER_USER_ID` 指定的用户）添加为文档协作者，权限设为 `full_access`（可管理）。使用 lark-cli 的文档成员添加能力（对应飞书 OpenAPI `POST /open-apis/drive/v1/permissions/:token/members`，`type = docx`），`member_type` 与 `member_id` 以 lark-cli 能力约定为准。添加成功后记录返回的 `member_id` 与角色。添加失败不阻断整体流程，仅在"五、面试官备注"末尾追加失败原因，不重试、不索取凭据。

### lark-cli 不可用

出现以下任一情况时停止写入飞书，并在对话输出的"五、面试官备注"末尾追加失败原因：

- lark-cli 未安装或不在 PATH
- 未登录、缺少文档创建、写入或权限管理权限
- 创建文档或写入块返回错误
- 返回结果中缺少 `document_id` 或文档链接

不要改用其他云文档产品，不要索取凭据，不要自动安装或授权。

### 脱敏

写入飞书前重新检查待写入内容，不写入手机号、身份证号、邮箱、具体住址、证件号、银行卡号等个人敏感信息。写入内容必须与对话输出一致。无法确认内容已安全脱敏时，停止写入并说明原因。

### 输出结果

写入成功后，在对话末尾追加：

- 飞书文档标题
- 文档链接
- 文档内题目总数与待办完成数（初始为 0）
- 六个固定分类各自的题目数量
- 协作者添加结果：是否成功、被添加的 `member_id`、权限角色（默认 `full_access`）；若失败则附上失败原因（与"五、面试官备注"末尾一致）

即使飞书写入失败或协作者添加失败，对话骨架输出仍按原规则呈现。

## 边界

- 支持粘贴文本、本地 PDF 与飞书消息附件三种输入；除飞书附件读取与飞书云文档写入外不访问其他外部系统，不持久化候选人信息。
- 收到简历且材料足以支撑后直接走完全流程，不得反问；仅在材料确实缺失必须阻断时才允许发问。
- 不得只回一段确认式回复；不得只输出非结构化的简历总结或面试建议。最终产出必须包含完整五段骨架与飞书云文档；第一次回复必须以 `## 一、候选人基本信息` 开头。
- PDF 读取失败时停止并说明原因，不自行 OCR，除非用户明确要求。
- 不回显个人敏感信息；写入飞书前必须再次脱敏。
- 面试题必须基于原始简历生成，不得基于总结后的版本生成；飞书文档内容必须与对话骨架完全一致。
- 输出骨架固定为五个一级标题（候选人基本信息 / 简历总结 / 岗位匹配度分析 / 面试题清单 / 面试官备注），题目清单固定为六个二级分类（`### 4.1` ~ `### 4.6`），不得调整、合并、省略或重命名；**严禁**用自由标题替换六个固定二级分类。
- 每道面试题在对话输出中必须使用 `- [ ] 题目` 待办复选框格式，考点缩进 2 空格单独成行；必问题在 `- [ ] ` 之后紧跟 `★`。不得写成 `1. 题目` 的编号列表形式。
- 默认按 3 分钟 / 题出题，默认 10 题 / 30 分钟；题目总数 = 用户指定时长 ÷ 3（四舍五入）。分类固定配额（默认 10 题）：4.1（含必问题）2 / 4.2 2 / 4.3 2 / 4.4 2（至少覆盖 2 个项目）/ 4.5 1 / 4.6 1。题目总数变化时优先在 4.2 / 4.3 / 4.4 之间增减，4.1 含必问题不得减到 1 以下，4.5 / 4.6 不得减到 0；配额用满即停。
- 用户显式指定单题时长或总时长时，按用户节奏重新计算总题数，再按上述缩放规则分配配额；未指定一律按 3 分钟 / 题、10 题 / 30 分钟执行。
- 每道面试题在飞书文档中使用一个独立待办块 `[]`，初始未勾选；考点不得放入待办块正文；必问题在待办块正文开头加 `★` 标记。
- 必问清单中的题目必须原样出现在所属分类最前面，不得省略、改写或与其他题目合并；新增或修改必问题只能在"必问清单"节内完成。
- 不输出参考答案、评分标准或追问清单，除非用户明确要求。
- 不生成编程笔试或限时作业题，除非用户明确要求。
- 不跨岗位方向混合出题；方向不确定时只向用户确认一次。
- 不编造原始简历中未出现的技术细节、数据或项目。
- lark-cli 不可用时不降级到其他云文档产品，不索取凭据，不自动安装或授权；保留对话骨架输出并在"五、面试官备注"末尾记录失败原因。
