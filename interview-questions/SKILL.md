# Interview Questions

按照"接收简历 → 识别岗位 → 提取考察点 → 生成题目 → 输出"的顺序处理。整个过程仅基于用户粘贴的简历文本，不读取本地文件、不访问外部系统、不持久化候选人信息。

## 接收简历

从当前对话中取得用户粘贴的简历全文。不要要求用户改为文件路径、下载链接或外部系统入口。

先穷尽已有内容。不要在未读完简历前发送信息收集模板或重复询问。

敏感信息只保留与考察相关的内容（姓名、教育、经历、技能、项目、成果）。遇到手机号、身份证号、邮箱、具体住址、证件号、银行卡号等个人敏感信息时，不在输出中回显，也不作为题目素材。若简历敏感信息过多导致无法继续，说明原因并停止。

简历明显残缺（例如只有姓名或只有一段经历）、无法支撑任何有意义的提问时，只请求取得当前阻断所需的最小补充（例如完整简历文本或期望岗位方向），不要求整份重写。

## 识别岗位方向

从岗位经历、技术栈、项目内容和教育背景推断候选人的主要方向：

- 后端 / 前端 / 客户端（iOS/Android/桌面） / 全栈
- 算法 / 机器学习 / 数据 / 数据工程
- 测试 / 测试开发 / 质量
- 运维 / SRE / 平台 / 安全
- 产品 / 项目管理 / 设计
- 其他（结合简历内容说明）

推断必须引用简历中的具体依据（例如"技术栈列出了 Java/Spring、多次项目涉及分布式事务"）。推断依据不充分时只向用户确认一次岗位方向，不要根据猜测继续出题。不要在一个清单中混合多个差异较大的岗位方向；当候选人经历横跨多个方向时，选择简历篇幅最集中、最近一段经历最相关的方向，并在输出顶部说明取舍。

## 提取考察点

从简历中抽取并归类以下内容，作为出题依据：

- 核心技能：简历中反复出现、用于关键项目或与岗位方向强相关的技术栈与能力
- 次核心技能：辅助性技术、工具、流程
- 项目亮点：明确写出的性能数据、业务结果、架构决策、从 0 到 1 的建设
- 可疑点或待澄清点：描述含糊、数据缺少口径、职责边界不清、时间线异常
- 成长轨迹：职级变化、带团队经历、跨团队协作

不要编造简历中未出现的信息，也不要把合理推断写成简历原文。

## 生成题目

按以下分类分组生成面试题。每个分类只生成与候选人简历强相关的题目，不输出通用题库式问题。

- **基础题**：考察核心技能的基本概念、原理和常用实现。贴近岗位方向。
- **进阶题**：考察底层机制、性能、边界、设计取舍，难度高于基础题。
- **项目深挖题**：针对简历中的具体项目，追问候选人的角色、方案选择、难点、关键数据、复盘。
- **场景 / 设计题**：结合候选人经历给出一个业务场景或系统设计问题，考察拆解和方案能力。
- **软素质题**：围绕协作、推动、复盘、抗压、目标管理等维度，必须引用简历中的具体段落。

岗位方向不同则侧重不同：工程类岗位加大基础 / 进阶 / 场景题比重；算法 / 数据类岗位在基础与进阶中侧重模型、数据、评估与工程落地；产品 / 项目管理岗位减少技术底层题，增加需求拆解、指标设计、跨团队协作与复盘题。

每个分类的题目数量根据简历信息密度动态调整，不硬编码固定数量。基础与进阶题合计不少于 6 题；项目深挖题覆盖简历中至少 2 个重要项目（如信息不足则减少并在输出中说明）；场景 / 设计题至少 1 题；软素质题至少 2 题。

不要在题目之外输出参考答案、评分标准或追问清单，除非用户明确要求。不要把题目写成编程笔试或限时作业。

## 输出

按以下顺序组织输出内容：

1. **候选人画像**
   - 识别到的岗位方向及推断依据
   - 简历关键亮点（3–5 条）
   - 建议重点考察的方向（2–4 条，结合可疑点或亮点）

2. **面试题清单**
   - 按分类分组：基础 / 进阶 / 项目深挖 / 场景或设计 / 软素质
   - 每题格式：`1. 题目` + 一行 `考点：…`
   - 项目深挖题在题目中点名对应的简历项目

3. **补充说明**
   - 简历中信息不足、无法覆盖的分类
   - 需要用户进一步确认的岗位方向或考察重点

全部题目使用统一编号，分类标题清晰，避免重复考查同一知识点。

## 边界

- 不读取本地文件，不访问外部系统，不持久化候选人信息。
- 不回显手机号、身份证号、邮箱、具体住址等个人敏感信息。
- 不输出参考答案、评分标准或追问清单，除非用户明确要求。
- 不生成编程笔试或限时作业题，除非用户明确要求。
- 不跨岗位方向混合出题；方向不确定时只向用户确认一次。
- 不编造简历中未出现的技术细节、数据或项目。
