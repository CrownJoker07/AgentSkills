# Interview Questions

按照"接收简历 → 总结简历 → 识别岗位 → 提取考察点 → 生成题目 → 输出到对话 → 写入飞书云文档"的顺序处理。整个过程基于用户提供的原始简历内容，不持久化候选人信息；飞书云文档仅通过 lark-cli 写入，不直接调用 OpenAPI。

## 接收简历

支持以下两种输入方式，优先使用用户已提供的材料，不要同时要求两种：

- **粘贴文本**：用户在当前对话中直接粘贴简历全文。不要要求用户改为文件路径、下载链接或外部系统入口。
- **本地 PDF 文件**：用户提供本地 PDF 文件路径。使用 Read 工具读取该 PDF，按页面顺序提取全文（包含文字、表格、列表）。读取后保留原文段落结构，不要丢弃看似重复或冗余的内容。

PDF 读取失败（路径不存在、非 PDF、加密、扫描型 PDF 无法提取文字等）时，说明具体原因并停止。不要把 PDF 转换为图片再 OCR，除非用户明确要求。当 PDF 无法处理时，只向用户请求最小替代（粘贴文本或换一份 PDF），不要求整份重写。

先穷尽已有内容。不要在未读完简历前发送信息收集模板或重复询问。

敏感信息只保留与考察相关的内容（姓名、教育、经历、技能、项目、成果）。遇到手机号、身份证号、邮箱、具体住址、证件号、银行卡号等个人敏感信息时，不在输出中回显，也不作为题目素材。若简历敏感信息过多导致无法继续，说明原因并停止。

简历明显残缺（例如只有姓名或只有一段经历）、无法支撑任何有意义的提问时，只请求取得当前阻断所需的最小补充（例如完整简历文本或期望岗位方向），不要求整份重写。

## 总结简历

在识别岗位和出题之前，先对原始简历做一份简要总结，便于用户快速核对候选人全貌。总结必须基于原始简历，不得引入外部信息或编造成就。

总结内容包括：

- 基本信息：姓名（若有）、教育背景、工作年限
- 岗位方向与当前定位（从经历推断）
- 主要工作经历（按时间倒序，每家公司的角色与核心职责）
- 代表性项目（2–4 个，写明项目目标、候选人的角色、使用的技术栈、关键结果）
- 核心技能栈
- 简历中的可量化成果（性能数据、业务指标、团队规模等）

总结只用于呈现给候选人与面试官快速对齐，**后续识别岗位、提取考察点和生成面试题必须基于原始简历全文，不得基于总结版本**。总结中省略的细节仍要保留在原始简历中供后续步骤使用。

## 识别岗位方向

从原始简历中的岗位经历、技术栈、项目内容和教育背景推断候选人的主要方向：

- 后端 / 前端 / 客户端（iOS/Android/桌面） / 全栈
- 算法 / 机器学习 / 数据 / 数据工程
- 测试 / 测试开发 / 质量
- 运维 / SRE / 平台 / 安全
- 产品 / 项目管理 / 设计
- 其他（结合简历内容说明）

推断必须引用原始简历中的具体依据（例如"技术栈列出了 Java/Spring、多次项目涉及分布式事务"）。推断依据不充分时只向用户确认一次岗位方向，不要根据猜测继续出题。不要在一个清单中混合多个差异较大的岗位方向；当候选人经历横跨多个方向时，选择原始简历篇幅最集中、最近一段经历最相关的方向，并在输出顶部说明取舍。

## 提取考察点

从原始简历中抽取并归类以下内容，作为出题依据：

- 核心技能：原始简历中反复出现、用于关键项目或与岗位方向强相关的技术栈与能力
- 次核心技能：辅助性技术、工具、流程
- 项目亮点：明确写出的性能数据、业务结果、架构决策、从 0 到 1 的建设
- 可疑点或待澄清点：描述含糊、数据缺少口径、职责边界不清、时间线异常
- 成长轨迹：职级变化、带团队经历、跨团队协作

不要编造原始简历中未出现的信息，也不要把合理推断写成简历原文。不要因为总结中已经概括过就跳过原始简历中的细节；总结中被省略的内容仍需重新在原始简历中扫描。

## 生成题目

按以下分类分组生成面试题。每个分类只生成与候选人原始简历强相关的题目，不输出通用题库式问题。题目必须能追溯到原始简历中的具体段落、项目或技术点，不得基于总结后的概括性描述生成。分类顺序固定为：动机与职业规划 → 技术基础 → 技术进阶 → 项目深挖 → 场景 / 设计 → 软素质，不得调整、合并或跳过任何分类（信息不足时在对应分类下注明"本分类无题，原因：…"）。

- **动机与职业规划**：考察候选人为什么来面试该岗位、对该岗位的理解、实习目标、毕业去向与中长期职业规划、对目标城市 / 行业 / 团队风格的偏好。必须结合原始简历中可推断的信息提问（例如"你过往项目集中在游戏方向，为什么本次选择 U3D 客户端实习"、"你对未来 3 年的技术成长路径是什么"），不要出与简历完全无关的通用题。
- **技术基础**：考察核心技能的基本概念、原理和常用实现。贴近岗位方向。
- **技术进阶**：考察底层机制、性能、边界、设计取舍，难度高于基础题。
- **项目深挖**：针对原始简历中的具体项目，追问候选人的角色、方案选择、难点、关键数据、复盘。
- **场景 / 设计**：结合候选人经历给出一个业务场景或系统设计问题，考察拆解和方案能力。
- **软素质**：围绕协作、推动、复盘、抗压、目标管理等维度，必须引用原始简历中的具体段落。

岗位方向不同则侧重不同：工程类岗位加大技术基础 / 技术进阶 / 场景题比重；算法 / 数据类岗位在技术基础与技术进阶中侧重模型、数据、评估与工程落地；产品 / 项目管理岗位减少技术底层题，增加需求拆解、指标设计、跨团队协作与复盘题。

每个分类的题目数量按以下固定下限生成，根据原始简历信息密度向上调整：动机与职业规划不少于 3 题；技术基础与技术进阶合计不少于 6 题；项目深挖覆盖原始简历中至少 2 个重要项目（如信息不足则减少并在输出中说明）；场景 / 设计至少 1 题；软素质至少 2 题。

不要在题目之外输出参考答案、评分标准或追问清单，除非用户明确要求。不要把题目写成编程笔试或限时作业。

## 输出

在对话中按以下固定骨架组织输出内容，作为用户核对内容以及写入飞书云文档的统一来源。骨架结构固定，每次生成必须严格包含以下一级标题，顺序、标题文字、子项顺序均不得调整或省略（信息不足时在对应子项下写"本项无内容，原因：…"）。

### 骨架

```text
# 面试题 - <候选人姓名或"未知"> - <岗位方向> - <YYYYMMDD>

## 一、候选人基本信息
- 姓名、学校、专业、年级 / 届别
- 期望岗位、期望城市、实习时间（原始简历有则列，无则标注"简历未提供"）
- 工作年限（在校 / 在职 / 离职）

## 二、简历总结
- 岗位方向与当前定位
- 主要经历（倒序，每段一行：时间 / 角色 / 核心职责）
- 代表性项目（2–4 个，每个：项目目标 / 角色 / 技术栈 / 关键结果）
- 核心技能栈
- 可量化成果（性能数据、业务指标、团队规模等）

## 三、岗位匹配度分析
- 岗位方向推断依据（引用原始简历）
- 关键亮点（3–5 条）
- 潜在风险点（2–4 条，结合可疑点、简历空缺或待澄清点）
- 建议重点考察的方向（2–4 条）

## 四、面试题清单

### 4.1 动机与职业规划
1. 题目
   考点：…
2. …

### 4.2 技术基础
…

### 4.3 技术进阶
…

### 4.4 项目深挖
…

### 4.5 场景 / 设计
…

### 4.6 软素质
…

## 五、面试官备注
- 原始简历中信息不足、无法覆盖的分类或子项
- 需要用户进一步确认的岗位方向或考察重点
- 建议追问方向（基于风险点给出 2–3 条）
```

### 约束

- 每个一级标题必须出现，顺序固定。
- 四类面试题的二级标题顺序固定（4.1 → 4.6），不得调换、合并或新增分类。
- 每道题目格式统一：`序号. 题目` + 换行 `考点：…`。
- 项目深挖题在题目正文开头点名对应的原始简历项目。
- 需要时标注该题对应的原始简历段落或项目名，便于回溯。
- 全部题目在每个分类内独立编号（每个分类从 1 开始）。
- 不在题目之外输出参考答案、评分标准或追问清单，除非用户明确要求。

## 写入飞书云文档

在完成对话输出后，使用 lark-cli 将同一份骨架内容（候选人基本信息 + 简历总结 + 岗位匹配度分析 + 面试题清单 + 面试官备注）写入飞书云文档。每道面试题使用待办块（`[]`），便于面试官在线上文档中勾选已完成项。

## 使用 lark-cli

优先使用已安装的 lark-cli Skills 与 `lark-docx` 能力创建和写入飞书云文档。能力名称或参数不确定时，先运行 `lark-cli schema` 或对应命令的 `--help`，再调用与飞书 OpenAPI 一一映射的文档接口；不要猜测快捷命令参数。

需要完整控制请求体时，使用 lark-cli Raw API。将 CLI 结果作为 JSON 处理：以退出码和顶层 `ok` 判断成功，再从 `data` 读取 OpenAPI 结果；不要要求顶层存在原始 OpenAPI 的 `code: 0`。

## 文档创建与写入

按以下步骤创建并写入文档：

1. 创建新文档：使用 lark-cli 的文档创建能力。文档标题统一为 `面试题 - <候选人姓名或"未知"> - <岗位方向> - <YYYYMMDD>`。记录返回的 `document_id` 与文档链接。
2. 获取文档根 block_id：通常是新文档的 `document_id` 或返回结构中的根块标识，具体字段以 lark-cli 返回为准。
3. 按顺序批量创建块，骨架固定如下（一级标题使用 `heading2`，二级分类标题使用 `heading3`，不得调整顺序、不得合并、不得省略）：
   - `heading2`："一、候选人基本信息"
   - `text`：姓名、学校、专业、年级 / 届别、期望岗位、期望城市、实习时间、工作年限
   - `heading2`："二、简历总结"
   - `text`：岗位方向与当前定位、主要经历、代表性项目、核心技能栈、可量化成果
   - `heading2`："三、岗位匹配度分析"
   - `text`：岗位方向推断依据、关键亮点、潜在风险点、建议重点考察的方向
   - `heading2`："四、面试题清单"
   - 六个固定二级分类，顺序固定：
     - `heading3`："4.1 动机与职业规划"
     - `heading3`："4.2 技术基础"
     - `heading3`："4.3 技术进阶"
     - `heading3`："4.4 项目深挖"
     - `heading3`："4.5 场景 / 设计"
     - `heading3`："4.6 软素质"
   - 每个分类下：
     - **每道面试题使用待办块 `todo`（`block_type = 17 / todo`），初始 `checked = false`**，块内首段为题目正文
     - 题目下方紧跟一个 `text` 块，内容为 `考点：…`，不塞进待办块正文
   - `heading2`："五、面试官备注"
   - `text`：信息不足的子项、需确认事项、建议追问方向
4. 一次批量请求中包含多块时按展示顺序排列；批量请求受 OpenAPI 单次上限限制时按顺序分批发送，保持原始顺序。

## 题目格式与勾选

- 每道题目对应一个独立的待办块，初始未勾选。
- 待办块的正文只放题目本身，不放"考点：…"，避免勾选时连考点一起被划掉影响阅读。
- 考点作为紧跟待办块的普通文本块，便于面试官对照。
- 项目深挖题的待办块正文开头保留对应原始简历项目名，便于在文档中检索。

## lark-cli 不可用

出现以下任一情况时停止写入飞书，并在对话输出的"补充说明"末尾追加失败原因：

- lark-cli 未安装或不在 PATH
- 未登录、缺少文档创建或写入权限
- 创建文档或写入块返回错误（记录具体错误类别）
- 返回结果中缺少 `document_id` 或文档链接

不要在飞书不可用时改用其他云文档产品，不要向用户索取凭据或应用配置，不要自动执行安装、登录或授权流程。

## 脱敏

写入飞书前重新检查待写入内容，不写入手机号、身份证号、邮箱、具体住址、证件号、银行卡号等个人敏感信息。写入内容必须与对话输出一致，不得额外扩展。无法确认内容已经安全脱敏时，停止写入并说明原因。

## 输出结果

写入成功后，在对话末尾追加：

- 飞书文档标题
- 文档链接（使用 lark-cli 返回的 URL）
- 文档内题目总数与待办完成数（初始为 0）
- 六个固定分类各自的题目数量

飞书文档为可选产物。即使飞书写入失败，对话中的骨架输出仍按原规则呈现。

## 边界

- 支持粘贴文本与本地 PDF 两种输入；除飞书云文档写入外不访问其他外部系统，不持久化候选人信息。
- PDF 读取失败时停止并说明原因，不自行转换格式或调用 OCR，除非用户明确要求。
- 不回显手机号、身份证号、邮箱、具体住址等个人敏感信息；写入飞书前必须再次脱敏。
- 面试题必须基于原始简历生成，不得基于总结后的版本生成；飞书文档内容必须与对话骨架输出完全一致。
- 输出骨架固定为五个一级标题（候选人基本信息 / 简历总结 / 岗位匹配度分析 / 面试题清单 / 面试官备注），题目清单固定为六个二级分类（4.1 动机与职业规划 / 4.2 技术基础 / 4.3 技术进阶 / 4.4 项目深挖 / 4.5 场景或设计 / 4.6 软素质），不得调整、合并、省略或重命名。
- 每道面试题在飞书文档中使用一个独立待办块 `[]`，初始未勾选；考点不得放入待办块正文。
- 不输出参考答案、评分标准或追问清单，除非用户明确要求。
- 不生成编程笔试或限时作业题，除非用户明确要求。
- 不跨岗位方向混合出题；方向不确定时只向用户确认一次。
- 不编造原始简历中未出现的技术细节、数据或项目。
- lark-cli 不可用时不降级到其他云文档产品，不索取凭据，不自动安装或授权；保留对话骨架输出并在"五、面试官备注"末尾记录失败原因。
