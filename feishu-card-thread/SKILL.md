# Feishu Card Thread

只负责飞书输入和输出通道。业务判断、诊断、修复和测试由当前任务适用的其他 Skill 或 Agent 完成。

本 Skill 在当前 Agent 已收到飞书消息或卡片交互后执行，不负责启动事件监听器。复用当前入口交付的事件，不另行启动事件订阅处理同一消息。

## 确认入口

从当前入站事件取得 `message_id`、`chat_id`、`thread_id` 和消息类型。卡片按钮回调只有在能够定位原卡片消息的 `message_id` 时才继续。

- 已有 `thread_id`：复用该话题，不再创建新话题。
- 没有 `thread_id`：以当前 `message_id` 作为根消息创建话题。
- 事件缺少 `message_id`：说明无法定位原消息并停止，不根据卡片文本搜索或猜测消息。

同一事件只处理一次。不要另建事件消费者、去重存储或任务状态模型。

## 使用 lark-cli

优先使用已安装的 lark-cli Skills 和 `lark-im` 能力。能力名称或参数不确定时，先运行 `lark-cli schema` 或对应命令的 `--help`，再调用与飞书 OpenAPI 一一映射的消息接口；不要猜测快捷命令参数。

需要完整控制请求体时，使用 lark-cli Raw API。将 CLI 结果作为 JSON 处理：以退出码和顶层 `ok` 判断成功，再从 `data` 读取 OpenAPI 结果；不要要求顶层存在原始 OpenAPI 的 `code: 0`。

若 lark-cli 不可用、未登录或缺少消息读写权限，报告具体检查结果并停止。不要输出凭据，也不要自动执行安装、登录、授权或修改飞书应用配置。

## 读取上下文

取得原始消息的 `message_id`，使用 lark-cli 读取：

- 当前卡片或消息的完整内容
- 引用消息和相关上下文
- 截图、录屏、日志和其他附件
- 已有话题回复和任务中已经提供的材料

先穷尽已有内容，不要求用户重复提供。下载任务所需附件，并将已有文字、附件路径和来源上下文交给业务处理方。不要判断根因或决定需要修改什么。

材料无法读取时说明具体失败原因，只请求取得当前阻断材料所需的最小补充，然后停止。

## 创建话题

原事件没有 `thread_id` 时，调用回复消息接口 `POST /open-apis/im/v1/messages/:message_id/reply` 回复原卡片消息，并显式提交 `reply_in_thread: true`、所需的 `msg_type` 和序列化后的 `content`。不得使用普通群消息发送能力代替。

从成功结果的 `data.thread_id` 记录话题 ID。即使当前只需确认收到，也必须通过这次话题回复确认。

原事件已有 `thread_id` 时直接记录并复用，不调用创建话题的回复。后续确认、问题、阶段进度和最终结果全部通过回复话题内消息或以 `thread_id` 为接收目标的受支持消息接口发送到该话题。

出现以下任一情况时说明阻断原因并停止：

- 没有原消息 `message_id`
- 可用能力不能设置 `reply_in_thread: true`
- 回复结果没有返回 `thread_id`
- 当前群聊不支持话题回复
- 飞书拒绝对当前消息类型创建话题

不得退化为群聊主消息流、私聊、新群消息或其他话题，也不得要求 webhook 预先提供尚未创建的 `thread_id`。

## 延续对话

将 `thread_id` 作为当前任务的飞书话题标识。收到后续消息时，确认事件的 `thread_id` 与当前话题一致，再把消息、附件和已有业务上下文交给同一业务处理方继续处理。

不要把其他 `thread_id` 的历史加入当前上下文。若当前运行时不能保持话题上下文隔离，说明该能力缺失并停止；不要在本 Skill 中实现持久化会话映射。

## 传递信息

将读取到的任务材料交给适用的业务 Skill。代码 BUG 使用 `repo-bugfix`；由其判断证据是否充足、是否需要补充以及应执行分析还是修复。

接收业务处理方提供的确认摘要、阻断问题、阶段进度和最终结果，并发送到已记录的 `thread_id`。不要自行扩展内容、改变结论或增加业务流程。

一次只发送当前必要的信息。缺少材料时只发送业务处理方确认的最关键问题，并说明已确认的事实和该信息为何阻断当前步骤。

## 脱敏

读取后识别凭据、个人数据、内部地址、私有代码、业务数据和其他敏感信息。只向业务处理方传递完成任务所需的最小材料。

每次发送飞书前重新检查并脱敏。不要发送凭据、完整内部路径、未经处理的日志、无关代码或附件内容。无法确认内容已经安全脱敏时，不通过飞书发送，并在当前会话说明阻断原因。

## 通道边界

- 只维护原卡片对应的一个 `message_id` 和一个 `thread_id`。
- 不监听飞书事件；只处理当前入口已接收并交付的事件。
- 不创建任务状态模型、持久化记录、调度器或固定 CLI 封装。
- 不诊断 BUG，不访问仓库，不修改、测试、提交或推送代码。
- 不在卡片话题之外发送任务信息。
