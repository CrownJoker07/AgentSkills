# Feishu Card Thread

只负责飞书输入和输出通道。业务判断、诊断、修复和测试由当前任务适用的其他 Skill 或 Agent 完成。

## 读取上下文

取得原始消息的 `message_id`，使用可用的飞书能力读取：

- 当前卡片或消息的完整内容
- 引用消息和相关上下文
- 截图、录屏、日志和其他附件
- 已有话题回复和任务中已经提供的材料

先穷尽已有内容，不要求用户重复提供。下载任务所需附件，并将已有文字、附件路径和来源上下文交给业务处理方。不要判断根因或决定需要修改什么。

材料无法读取时说明具体失败原因，只请求取得当前阻断材料所需的最小补充，然后停止。

## 创建话题

首次回复原卡片消息时，调用飞书回复消息能力并显式设置 `reply_in_thread: true`。不得使用普通群消息发送能力代替。

从回复结果记录 `thread_id`，后续确认、问题、阶段进度和最终结果全部发送到该 `thread_id`。即使当前只需确认收到，也必须通过这次话题回复确认。

出现以下任一情况时说明阻断原因并停止：

- 没有原消息 `message_id`
- 可用能力不能设置 `reply_in_thread: true`
- 回复结果没有返回 `thread_id`
- 当前群聊不支持话题回复

不得退化为群聊主消息流、私聊、新群消息或其他话题，也不得要求 webhook 预先提供尚未创建的 `thread_id`。

## 传递信息

将读取到的任务材料交给适用的业务 Skill。代码 BUG 使用 `repo-bugfix`；由其判断证据是否充足、是否需要补充以及应执行分析还是修复。

接收业务处理方提供的确认摘要、阻断问题、阶段进度和最终结果，并发送到已记录的 `thread_id`。不要自行扩展内容、改变结论或增加业务流程。

一次只发送当前必要的信息。缺少材料时只发送业务处理方确认的最关键问题，并说明已确认的事实和该信息为何阻断当前步骤。

## 脱敏

读取后识别凭据、个人数据、内部地址、私有代码、业务数据和其他敏感信息。只向业务处理方传递完成任务所需的最小材料。

每次发送飞书前重新检查并脱敏。不要发送凭据、完整内部路径、未经处理的日志、无关代码或附件内容。无法确认内容已经安全脱敏时，不通过飞书发送，并在当前会话说明阻断原因。

## 通道边界

- 只维护原卡片对应的一个 `message_id` 和一个 `thread_id`。
- 不创建任务状态模型、持久化记录、调度器或固定 CLI 封装。
- 不诊断 BUG，不访问仓库，不修改、测试、提交或推送代码。
- 不在卡片话题之外发送任务信息。
